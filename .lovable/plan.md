## What we know

- Every request to `/` returns 500 with `{"unhandled":true,"message":"HTTPError"}`.
- Server logs only show `TypeError: Class extends value [object Module] is not a constructor or null` — no file, no stack.
- No `class … extends …` exists anywhere in `src/`, so the failing `extends` lives inside a bundled dependency that is being handed a module namespace object instead of a class. That only happens when a source file introduces a bad import shape (namespace import used as a value, or a package pulled into the SSR bundle that ships CJS/ESM interop bugs on Cloudflare Workers).
- Recent turns changed only two things that reach the SSR root render:
  1. `src/components/HowItWorksModal.tsx` was fully overwritten (1658 lines) — imported by `src/routes/index.tsx` (the "/" route).
  2. `src/routeTree.gen.ts` had a `declare module` block removed and later manually re-added.

The new `HowItWorksModal.tsx` uses `React.ReactNode` and `React.CSSProperties` at lines 455, 514, 555, 876, 934 without importing React. Under TanStack Start's SWC transform + `verbatimModuleSyntax`, an unqualified `React.*` reference can be emitted as a runtime `React` identifier resolved via the automatic JSX runtime helpers — which in production Worker builds hands back the module namespace object of `react`, not a class. That's the exact shape that produces "Class extends value [object Module]" when downstream code touches it. It also explains why the crash only started after the modal overwrite.

## Fix

### Step 1 — Restore proper React imports in `HowItWorksModal.tsx`

Add an explicit type-only import so the `React.*` type references resolve at typecheck time and get fully stripped from the runtime bundle:

```ts
import type { ReactNode, CSSProperties } from "react";
```

Then replace every `React.ReactNode` → `ReactNode` and every `React.CSSProperties` → `CSSProperties` in the file (5 occurrences total).

### Step 2 — Stop hand-editing `src/routeTree.gen.ts`

The `declare module '@tanstack/react-start'` block that was manually re-added at the bottom of `src/routeTree.gen.ts` is regenerated by the TanStack Router Vite plugin. Manual edits get clobbered on every dev build and can drift from what the plugin expects. Remove the manual tail block; if a `Register` interface override is genuinely needed, it belongs in a new `src/router-register.d.ts` file that we control.

### Step 3 — Verify

- Reload `/` and confirm it renders instead of 500.
- Tail worker logs; the specific `TypeError` should be gone.

If the 500 still reproduces after Step 1, the wrapper in `src/server.ts` already logs `consumeLastCapturedError()`, but the current worker logs show only the generic message. As a follow-up we would widen `error-capture.ts` to also record synchronous throws during module init (`process.on('uncaughtException')` equivalent in the Worker via `addEventListener('error', …)` — already present) and add a one-line `console.error('[ssr-boot] loaded root route')` in `src/server.ts` so we can tell whether the failure is at import time or render time.

## Files touched

- `src/components/HowItWorksModal.tsx` — add `import type { ReactNode, CSSProperties } from "react"`; rename 5 `React.*` references.
- `src/routeTree.gen.ts` — delete the manually appended `declare module` tail block (lines after `._addFileTypes<FileRouteTypes>()`).

No dependency changes, no schema changes, no behaviour changes to any user-facing feature.
